## 摘要
随着高校学生消费观念由“购买新物”逐步转向“按需使用与循环利用”，校园二手交易需求明显增长。传统线下交易方式主要依赖熟人转发、班级群发布和临时摆摊，存在信息分散、交易时效差、沟通链路长等问题。针对上述痛点，本文设计并实现了一套校园二手交易微信小程序系统。系统采用“微信小程序 + 管理后台 + Spring Boot 后端”的整体方案，后端基于 MyBatis-Plus 与 MySQL 完成业务数据存储与访问，覆盖用户、商品、订单、收藏、评论、论坛、资讯、地址、咨询等核心功能。

本课题按照任务书和开题报告中的目标推进，重点完成了需求分析、总体设计、关键模块实现、系统联调测试和部署优化。在实现过程中，项目坚持“先打通主流程，再逐步优化体验”的策略，先保证业务闭环可运行，再进行样式兼容、云端构建稳定性等工程优化。最终结果表明，系统能够满足校园二手交易场景下的基本业务需求，具备较好的可用性与可扩展性。

关键词：校园二手交易；微信小程序；Spring Boot；MyBatis-Plus；前后端分离

## Abstract
With the growing need for low-cost and sustainable consumption on campus, second-hand trading among university students has become increasingly common. Traditional offline methods are often fragmented and inefficient. To solve these issues, this thesis designs and implements a campus second-hand trading system based on WeChat Mini Program, a web admin console, and a Spring Boot backend.

The system supports major functions including user management, product publishing and browsing, collection, commenting, order processing, forum interaction, news browsing, address maintenance, and merchant consultation. Following the objectives defined in the proposal and task book, this work completes requirement analysis, system design, implementation, testing, and deployment optimization. The final system demonstrates practical usability and extension potential for campus scenarios.

Key Words: Campus Second-hand Trading; WeChat Mini Program; Spring Boot; MyBatis-Plus; Front-end/Back-end Separation

## 第1章 绪论

### 1.1 研究背景
高校是二手交易最活跃的场景之一。每学期末或毕业季，大量书籍、电子设备、生活用品会集中流转。实际调研中可以发现，很多同学并不是没有交易需求，而是“找不到合适的交易对象”或“沟通成本太高”。线下交易虽然直接，但信息半径小、时效短、记录性差，一旦聊天记录丢失或信息沉底，交易线索很难持续跟进。

微信小程序天然适合这一场景：入口轻、上手快、不需要安装，用户可在碎片化时间内完成浏览、咨询和下单。基于这个出发点，本文选择以微信小程序作为用户端核心载体，并配套后台管理系统和后端服务，形成可持续运营的校园交易平台。

### 1.2 研究目的与意义
本课题的直接目标是实现一个“能真实跑起来”的校园二手交易系统，而不是仅停留在演示层面。系统要解决的核心问题包括：
- 信息聚合：把分散在群聊、朋友圈、线下口口相传的商品信息集中管理；
- 流程闭环：实现“发布-浏览-咨询-下单-管理”的完整链路；
- 运营支撑：提供后台管理能力，支持内容维护与数据治理；
- 工程落地：系统应可部署、可维护、可迭代。

从实际价值看，该系统可以降低学生交易门槛，提升闲置物品流通效率；从学习价值看，课题覆盖需求分析、架构设计、数据库设计、前后端联调、测试部署等完整软件工程流程。

## 第2章 相关技术与开发环境

### 2.1 微信小程序前端技术
微信小程序作为用户端，主要承载商品浏览、论坛互动、订单确认、个人中心等高频功能。相比传统 App，小程序更适合毕业设计阶段快速迭代：开发门槛较低、发布体验闭环快、用户触达路径短。项目中通过页面路由和组件复用构建主流程，保证了功能完整性和交互连贯性。

### 2.2 Spring Boot 后端框架
后端采用 Spring Boot，主要考虑两点：一是生态成熟，便于快速搭建业务接口；二是结构清晰，适合毕业设计中的分层实现。系统按 Controller、Service、DAO、Entity 进行组织，降低模块耦合度，便于后续维护和功能扩展。

### 2.3 MyBatis-Plus 与 MySQL
在数据访问层，使用 MyBatis-Plus 可以减少样板代码，提高 CRUD 开发效率。数据库选择 MySQL，原因是部署成本低、使用广泛、与当前技术栈兼容性好。系统核心业务表围绕用户、商品、订单、评论、收藏、论坛、资讯与咨询展开。

### 2.4 管理端技术栈
管理端使用 Vue2 + Element UI。该组合在后台管理场景中具备组件完善、开发效率高的优势。通过统一页面风格和模块化菜单，管理员可快速完成商品维护、订单处理、资讯管理与字典配置等工作。

### 2.5 开发环境与工程实践
项目采用 Maven 管理后端依赖，前端构建后将产物纳入后端静态资源目录。为应对本机构建资源限制，项目增加了云端构建方案，配合代码托管平台完成自动化打包，提升了构建流程的稳定性与复现性。

## 第3章 需求分析

### 3.1 角色分析
系统主要涉及两类角色：
- 普通用户：关注商品发布、浏览、咨询、下单和个人信息维护；
- 管理员：关注内容治理、数据维护、交易过程管理与系统配置。

这种角色划分的好处是职责明确：用户侧强调“便捷交易”，管理侧强调“可控运营”。

### 3.2 业务流程分析
校园二手交易核心流程可概括为：
1. 用户注册登录并完善信息；  
2. 浏览商品列表，查看商品详情；  
3. 对意向商品发起咨询或收藏；  
4. 进入订单确认流程；  
5. 管理端跟踪与维护相关数据。

在这个流程中，“咨询”是一个关键节点。很多校园交易并非标准电商“直接下单”，而是先沟通再决定，因此系统专门设计了咨询相关模块来降低沟通摩擦。

### 3.3 功能需求分析
#### 3.3.1 小程序端
- 账号相关：注册、登录、找回密码、个人中心；
- 商品相关：商品发布、商品列表、商品详情、收藏、评论；
- 交易相关：订单确认、订单查询；
- 社区相关：论坛发帖回帖、资讯浏览；
- 其他：地址管理、咨询沟通。

#### 3.3.2 管理端
- 用户、商品、订单、评论、收藏管理；
- 论坛与资讯管理；
- 字典与配置管理；
- 咨询记录查看与维护。

#### 3.3.3 后端
- 提供统一业务接口；
- 处理核心业务校验与数据一致性；
- 提供文件、配置、字典等通用服务能力。

### 3.4 非功能需求分析
- 可用性：界面操作路径应简短，学习成本低；
- 稳定性：主流程可持续运行，避免“能演示但不可用”；
- 可维护性：代码结构清晰，便于后续补充新功能；
- 安全性：具备基础鉴权与输入校验。

## Abstract
This thesis focuses on campus second-hand trading scenarios and proposes a complete information system to address common issues such as scattered information, low transaction efficiency, high communication cost, and weak operational management. The system consists of a WeChat Mini Program client, a web-based administration module, and a Spring Boot backend service. MyBatis-Plus and MySQL are used for data persistence.

Aligned with the task-book and proposal objectives, this work covers requirement analysis, architecture design, database modeling, core function implementation, system testing, and deployment optimization. Results show that the system can support major campus second-hand trading workflows and provides practical value for university idle-resource circulation.

Key Words: Campus Second-hand Trading; WeChat Mini Program; Spring Boot; MyBatis-Plus; Admin System

## 第1章 绪论

### 1.1 选题背景
在高校环境中，教材、电子产品、生活用品等闲置物品更新快、流通需求强。传统线下交易方式依赖熟人圈或临时群消息，存在信息不对称、沟通链路长、交易过程缺乏沉淀管理等问题。微信小程序具备轻量、触达快、使用门槛低的优势，适合校园高频低客单价的二手交易场景。

### 1.2 研究目的与意义（对应开题/任务书）
本课题旨在构建一套可落地的校园二手交易系统，实现“发布-浏览-咨询-下单-管理”的闭环流程，具体意义如下：
- 为在校学生提供便捷可靠的二手交易渠道；
- 提高闲置物品流通效率，降低资源浪费；
- 通过后台管理提升内容治理与运营效率；
- 为后续扩展信用评价、推荐算法等功能提供系统基础。

### 1.3 主要研究内容（任务分解）
1. 完成系统需求分析与角色权限划分；  
2. 完成总体架构与数据库结构设计；  
3. 实现微信小程序端核心业务页面；  
4. 实现管理端数据管理与运营功能；  
5. 实现后端接口、业务逻辑与数据访问；  
6. 进行系统测试并完成阶段性优化。

## 第2章 相关技术与开发环境

### 2.1 微信小程序前端
小程序端作为用户主要入口，承担登录注册、商品浏览、订单操作、论坛互动、个人中心等功能。项目已配置多页面路由，覆盖主要业务路径，适合移动端轻交互场景。

### 2.2 后端技术栈
后端采用 Spring Boot 构建 REST 服务，以 Controller/Service/DAO 分层组织代码。数据访问层使用 MyBatis-Plus，数据库使用 MySQL，实现业务数据的结构化存储与查询。

### 2.3 管理端技术栈
管理端采用 Vue2 + Element UI，面向管理员进行商品、订单、用户、资讯、论坛、字典配置与咨询信息管理。前后端通过接口联调，实现统一业务逻辑。

### 2.4 工程构建与部署
后端使用 Maven 打包，前端构建产物可集成到后端静态资源目录。项目已实践 GitHub Actions 云端构建流程，用于提高管理端打包的稳定性和可重复性。

## 第3章 需求分析

### 3.1 角色与业务需求
系统主要角色为普通用户、管理员（后台运营），核心需求如下：
- 普通用户：账号管理、商品浏览与发布、收藏评论、订单确认、论坛互动、地址维护、咨询沟通；
- 管理员：用户数据管理、商品与订单管理、内容管理、字典与系统配置管理、咨询记录管理。

### 3.2 功能需求
#### 3.2.1 小程序端功能
- 登录、注册、忘记密码、个人信息维护；
- 商品列表/详情、商品发布、收藏、评论；
- 订单确认与订单列表查询；
- 论坛发帖回帖、资讯浏览；
- 地址管理与个人中心管理。

#### 3.2.2 管理端功能
- 用户、商品、订单、评论、收藏管理；
- 资讯与论坛内容管理；
- 字典与配置管理；
- 商家咨询数据管理。

#### 3.2.3 后端功能
- 提供统一 API 与参数校验；
- 支持主要业务实体的数据读写；
- 提供文件、配置、字典等公共能力。

### 3.3 非功能需求
- 可用性：流程简洁，便于学生快速上手；
- 可维护性：模块划分清晰，便于后续扩展；
- 稳定性：支持本地与云端构建部署；
- 安全性：具备基础鉴权和数据校验机制。
